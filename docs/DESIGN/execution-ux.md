# 执行确认、权限与修复体验设计

> 目标：把“AI 能执行”收敛成一套统一、可理解、可审计、可降级的产品链路。
> 范围包含：任务确认、工具确认、运行时权限请求、授权作用域、执行预览、可操作错误与预定义修复动作。

## 设计边界

这份文档只关心**跨模块交互设计与约束**，不写死实现细节。接口与字段以 `src/` 为准。

核心边界：

1. **高风险动作必须先确认**
2. **授权必须有清楚的资源范围**
3. **拒绝或超时后必须降级，而不是继续硬执行**
4. **修复动作优先走白名单 action，不直接演化成任意命令执行**

## 机制分层

| 机制 | 粒度 | 时机 | 作用 |
|------|------|------|------|
| `execute_plan` / 任务确认 | 任务级 | 执行前 | 让用户先理解“接下来要做什么” |
| `confirmRequired` | 工具级（静态） | 工具执行前 | 高危工具一律确认 |
| `permission_request` | 请求级（动态） | 执行前 | 对越界资源或首次敏感行为发起授权 |
| `actionable_error` / repair action | 失败后补救 | 执行失败后 | 把“报错”升级成“下一步可执行动作” |

## 当前已实现

- **WebChat**
  - 确认弹窗支持 `once / session / persistent`
  - 展示执行预览
  - 任务工作台可查看当前任务、审计回放、授权中心、工作区设置
- **钉钉**
  - 支持文本确认：`允许 <id>` / `允许 本会话 <id>` / `允许 永久 <id>` / `拒绝 <id>`
- **授权复用**
  - 持久授权落到 SQLite
  - Web 控制面可查看当前会话授权与长期授权
- **当前内置工具边界**
  - `read_file / list_files`：默认允许 `tools.fileBase` 内相对路径；越界绝对路径走 `permission_request`
  - `write_file`：仅允许写 `tools.fileBase` 内相对路径；走 `confirmRequired` + 执行预览
  - `browse_url / search_web`：访问外部网页或联网搜索前会先请求权限

## 推荐的用户链路

### 1. 任务确认

当模型准备发起多步任务时，应优先给出：

- 任务标题
- 计划步骤
- 风险摘要
- 将访问的资源范围
- 为什么需要这些资源

用户先决定“是否让这次任务继续”。

### 2. 资源授权

当某一步需要额外资源时，再发起 `permission_request`。

授权应始终附带：

- `action`：要做什么
- `message`：为什么需要
- `preview`：会访问哪些路径、域名、数据库或动作
- `scopeOptions`：`once / session / persistent`
- `grantKey`：可复用授权的资源键

### 3. 工具确认

对写入、删除、执行命令等高危动作，即使已经有计划级确认，也仍可要求工具级确认。

原因：

- 计划确认解决“知情”
- 工具确认解决“临门一脚的安全兜底”

### 4. 拒绝后的降级

当用户拒绝、超时或渠道不支持确认时，运行时必须降级为：

- 只给方案
- 解释未执行原因
- 保留任务上下文与审计记录

不能继续假装已经完成执行。

## 资源粒度建议

### 文件

- 以目录或路径范围建模
- `fileBase` 内相对路径可做低风险默认放行
- 越界绝对路径必须显式授权

### 网络

- 以域名或域名集建模
- 搜索引擎与具体网页访问应分别建模

### 数据库

- 以连接 + 库 + schema + 表/列白名单建模
- 默认只读

### 修复动作

- 以预定义 action 建模，例如 `install_chromium`
- 不与自由命令执行混用

### 受限命令

- 以 `shell / cwd / command` 作为预览核心
- 仅 owner
- 始终确认
- 超时、输出截断、禁止交互和常驻进程

## 可操作错误与修复动作

目标：把“去终端手抄命令”升级成“用户能理解并一键修复”。

典型场景：Chromium 未安装。

推荐分层：

| 层级 | 职责 |
|------|------|
| 工具层 | 返回结构化错误或 repair action 建议 |
| Gateway / Runtime | 验证动作是否合法、是否需要确认、如何执行与回传进度 |
| 渠道层 | 有按钮则展示按钮；无按钮则降级为文本说明 |

推荐原则：

- 优先做**白名单 repair action**
- 不要因为要修一个已知问题，就引入泛化 `exec` 入口
- 修复动作同样要进入审计链路

## 渠道降级策略

| 渠道能力 | 策略 |
|----------|------|
| 有按钮、有面板 | 展示确认面板、修复动作按钮、执行预览 |
| 只有按钮 | 压缩为按钮 + 简短说明 |
| 无按钮 | 降级为文本确认协议 |

现状：

- WebChat：强交互主界面
- 钉钉：文本确认可用，后续可继续补卡片化交互

## 三层执行路线约束

后续执行能力建议严格分层推进：

1. **预定义 repair action**
2. **受限 `run_command`**
3. **真正代码沙箱**

不要把三者混成一个“大而乱”的 exec 能力，否则权限模型、审计、失败兜底都会迅速失控。

## 下一步重点

- 把确认弹窗升级成更强的任务确认与执行摘要体验
- 把授权中心做得更清楚：展示、撤销、复用范围解释
- 为数据库工具补齐资源建模、只读默认和脱敏约束
- 把 repair action 纳入统一的任务确认与审计链路
