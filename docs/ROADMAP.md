## 当前主线：Phase 2a

当前主轴不是继续铺渠道，而是把“风险可控的本地执行助手”这条线收扎实。

判断标准：

- 没有清楚的权限边界、执行预览和审计回放，“能干活”就会退化成“能闯祸”
- 本地 Web 控制面是一等入口，聊天渠道主要承担触发、通知和轻交互
- 新增工具必须服从同一套确认、授权、审计模型

## 已完成摘要

### Phase 0 + Phase 1

- 项目规划与脚手架完成：TypeScript + Node.js + pnpm + Fastify + Vitest
- WebChat 跑通纯对话闭环，支持多轮上下文与流式输出
- 钉钉 Stream 渠道落地，不需要公网 IP
- 模型路由、Failover、费用估算、`onboard` / `doctor` CLI 已上线
- **V1（v0.1.0）已发布**：WebChat + 钉钉纯对话

### 已落地的 Phase 2a 基础能力

- WebChat Token 认证
- SQLite 对话持久化、历史分页、滑动窗口
- Function Calling + Owner 权限
- 基础内置工具：`get_current_time`、`browse_url`、`search_web`、`read_file`、`list_files`、`write_file`
- `confirmRequired`、`permission_request`、授权作用域、执行预览、拒绝后降级、审计回放
- WebChat 工作台：当前任务、审计回放、授权中心、工作区设置
- 工作区人格化：`AGENT.md / USER.md / IDENTITY.md / SOUL.md`

## Phase 2a：正在收口的工作

### 2a.1 权限模型与风险边界

- [ ] 把授权从“是否允许执行”继续收口到更明确的**资源粒度**
  - 文件 / 目录范围
  - 域名范围
  - 后续数据库、repair action、受限命令的范围模型
- [ ] 权限中心支持更清楚的**展示 / 撤销 / 复用说明**
- [ ] WebSocket 连接数限制与入口级限流配置化
- [ ] 为后续数据库工具预留结果脱敏、只读默认、列白名单边界

### 2a.2 任务确认与执行摘要体验

- [ ] 把当前“确认弹窗”升级成更强的**任务确认面板**
- [ ] 更清楚地展示：
  - 任务标题
  - 风险摘要
  - 即将访问的资源
  - 已覆盖的授权范围
  - 步骤级执行状态
- [ ] 继续把执行错误做成**可操作**的下一步，而不是只报错

详细设计见 [docs/DESIGN/execution-ux.md](./DESIGN/execution-ux.md)。

### 2a.3 执行底座与 2b 交接准备

- [ ] 把现有工具链路和任务确认、权限模型、审计回放继续打通
- [ ] 明确哪些能力属于 2a 收口，哪些正式进入 2b 闭环
- [ ] 为浏览器深化、数据库工具、Skills 等 2b 能力预留一致的权限与审计接口

### 2a.4 受限执行路线

这条线必须分层推进，不能一开始做成泛化 `exec`：

- [ ] **预定义 repair action**：例如安装 Chromium、已知修复动作
- [ ] **受限 `run_command`**：仅 owner、始终确认、展示 shell / cwd / command、超时、输出截断、禁止交互和常驻进程
- [ ] **真正代码沙箱**：与宿主机命令执行分层设计，不混成一个工具

相关约束见 `DEC-026`、`DEC-034`、`DEC-035`。

### 2a.5 仍在队列中的旧待办

这些仍有价值，但不是当前最优先：

- [ ] API 响应缓存
- [ ] 对话搜索与导出
- [ ] Token 预算裁剪与摘要压缩
- [ ] 配置热加载

## 后续阶段（保留规划宽度，但不承诺近期排期）
